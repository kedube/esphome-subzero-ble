== ESPHome Commands

All commands assume you've activated the Python virtual environment:

[source,bash]
----
source esphome-venv/bin/activate
----

=== Compile (Build Without Flashing)

[source,bash]
----
esphome compile subzero.yaml
----

=== Flash via USB

Connect your ESP32 via USB, then compile and flash in one step:

[source,bash]
----
ls /dev/cu.*    # find your device name (e.g. /dev/cu.usbserial-210)
esphome run subzero.yaml --device /dev/cu.usbserial-210    # macOS
----

This will compile, flash, and immediately show logs.

=== Flash Over-the-Air (OTA)

Once the ESP32 is on your network and running ESPHome, you can flash wirelessly:

[source,bash]
----
esphome run subzero.yaml --device 192.168.x.x --no-logs
----

Replace `192.168.x.x` with your device's IP address.

=== View Logs

.Via USB
[source,bash]
----
esphome logs subzero.yaml --device /dev/cu.usbserial-210
----

.Via network
[source,bash]
----
esphome logs subzero.yaml --device 192.168.x.x
----

[[ble-stack-tuning]]
== BLE Stack Tuning (sdkconfig_options)

The ESP-IDF BLE stack has several compile-time settings that affect reliability and connection limits.
These are set in your device YAML under `esp32: framework: sdkconfig_options:`.

=== Required for All Configs

[source,yaml]
----
esp32:
  board: esp32dev
  framework:
    type: esp-idf
    sdkconfig_options:
      CONFIG_BT_GATTC_CACHE_NVS_FLASH: "n"
----

Disables persisting the GATT client cache to flash.
Without this, stale cached GATT data causes D5 characteristic discovery failures after re-pairing or bond changes.
This is already set in all example configs.

=== Required for 2+ Concurrent Appliances

If you are connecting to *two or more* appliances from a single ESP32, add these options:

[source,yaml]
----
esp32:
  board: esp32dev
  framework:
    type: esp-idf
    sdkconfig_options:
      # Disable GATT cache persistence - stale cached data causes D5 discovery
      # failures after re-pairing. Required for all configs.
      CONFIG_BT_GATTC_CACHE_NVS_FLASH: "n"
----

WARNING: These options increase RAM usage in the BLE stack. An ESP32-S3 with PSRAM can handle more.

If you only have *one* appliance, you do not need these - the defaults work fine.

== Memory Considerations

The ESP32 (non-PSRAM) has only 320 KB of RAM. With BLE client connections, the BLE stack alone consumes a significant portion.
This integration uses several strategies to avoid out-of-memory crashes:

* JSON response buffers are *pre-reserved* to 2048 bytes at startup to avoid repeated heap reallocation during the 40+ BLE indication fragments, and the buffer's allocation is retained across messages
* Complete messages are parsed *in place* out of the reassembly buffer (no copy-out), using ArduinoJson's zero-copy mode - parsed strings point into the buffer instead of being duplicated, roughly halving peak parse memory; the temporary JSON document is freed immediately after parsing
* JSON parsing runs on the *main loop* (via the hub component), not inside the BLE notification callback
* Unchanged sensor and text-sensor values are *not re-published* on each 60s poll - this suppresses dozens of redundant Home Assistant API messages per poll cycle. Side effect: `last_updated` on static entities (model, serial, firmware versions) only advances when the value actually changes.
* Logger levels are tuned to suppress high-frequency BLE debug messages that would otherwise overwhelm the API frame buffer

If you experience crashes (typically `__cxa_allocate_exception` -> `operator new` -> `abort()`), try:

. Reducing the number of Home Assistant API connections to the device
. Setting `ble_client: WARN` in the logger config
. Lengthening the poll interval (default `60s`, set in the hub) to `120s` or `300s`

[[reconnect-logic]]
== Reconnect Logic

The integration runs a periodic poll every 60 seconds against each bonded appliance and keeps things alive with two simple mechanisms: a scheduled session refresh (pre-empts the firmware's unlock-session expiry) and stale-bond cleanup (catches an overnight bond timeout).

=== Scheduled Session Refresh

Roughly 18 minutes after each successful unlock (`kSessionRefreshIntervalMs`), the integration proactively disconnects and lets the fast-reconnect path re-establish the session - cached GATT handles, re-encrypt, re-subscribe, fresh unlock, fresh full-state poll. This pre-empts the appliance firmware's own ~20-25 minute BLE unlock-session expiry, which would otherwise kick the connection unpredictably. The refresh is tagged as intentional so it never counts toward stale-bond detection.

=== Silent-Poll Diagnostics (`poll_miss`)

`poll_ok` is set to `true` only when the parser reports a successful *poll response* (`{"status":0,"resp":{...}}`) - push notifications never count. Each `periodic_poll` clears `poll_ok` and writes both `unlock_channel` and `get_async` to D6; if no poll response arrived by the next tick, `poll_miss` increments and shows up in the logs as `miss=N`.

IMPORTANT: `poll_miss` is *diagnostic-only* - it does not trigger a reconnect. Earlier releases reconnected after 3 misses ("zombie detection"); that was replaced by the scheduled session refresh, which achieves the same recovery on a predictable cadence without churning healthy push-streaming connections.

Two situations produce a climbing `miss` counter:

* **fw 8.5 silent-poll behavior.** On Sub-Zero fridges and wall ovens running fw 8.5 / API 5.4, the second-and-later `get_async` writes within a held-open connection go silent - only a full BLE disconnect/reconnect produces a working session again. The official Sub-Zero Android app handles this the same way (verified via HCI snoop). Net effect on fw 8.5: roughly one full state refresh per ~18-minute session-refresh cycle, plus continuous push notifications in between.

* **Zombie unlock session.** If the bond or unlock state goes bad appliance-side (e.g. after the appliance re-pairs with the official app), the connection stays up but every poll goes silent - `miss` climbs indefinitely and no data ever parses. Session refreshes won't recover a broken bond; if `miss` keeps climbing across multiple refresh cycles, press *Reset Pairing*, power-cycle the appliance, and re-pair.

=== Boot Stagger via `poll_offset`

`poll_offset` (config field per appliance, default `0s`) is applied at the top of `periodic_poll`, `post_bond`, and `fast_reconnect`. Setting per-appliance values like `0s / 5s / 10s` on a multi-appliance ESP32 spreads the bonding chains in time so the concurrent-bonding race doesn't trigger in the first place. The scheduled session refresh remains as a safety net for sessions that come up broken.

=== Log Format

[source]
----
[I][szg:XXX]: [Left Dishwasher] Periodic poll D6 (miss=0, retries=0)
[I][szg:XXX]: [Wall Oven]      Periodic poll D6 (miss=2, retries=0)
[E][szg:XXX]: [Kitchen Range]  Pairing rejected (status 302). The PIN has likely changed - press 'Start Pairing' on the appliance and enter the new PIN in HA.
----

* `miss` - silent-poll counter (diagnostic-only; climbs during fw 8.5 silent windows, resets on the next successful poll response)
* `retries` - fast-reconnect retry counter (0–3, bond clear at 3; see below)
* `Pairing rejected (status 302)` - the appliance refused `unlock_channel`, usually because its PIN rotated after a power cycle. Press *Start Pairing* and re-enter the PIN displayed on the appliance.

=== Fast Reconnect + Bond Clear

Separately, when the BLE connection drops and `auto_connect: true` reconnects, the integration tracks `fast_retries`. If the reconnect fails three times in a row (common symptom: stale bond after an overnight timeout), the integration calls `esp_ble_remove_bond_device()`, clears all cached characteristic handles, and re-pairs from scratch on the next connect.

=== Tuning

* `interval: 60s` in the hub - poll cadence. Shortening to 30s doubles radio utilization (still single-digit percent for typical configs).
* `kSessionRefreshIntervalMs` (18 min, in `hub.h`) - session-refresh cadence. Must stay below the firmware's ~20-25 min unlock-session expiry.
* `poll_offset` per appliance - boot stagger. `0s` for first, `5s` or `10s` for subsequent appliances on the same ESP32.

[[parser-component]]
== Parser Component

The JSON parsing logic (poll responses + push notifications from all three appliance types) lives in a small external C++ component at `components/subzero_protocol/`, rather than inside the YAML lambdas. This keeps the lambdas thin and - more importantly - makes the parser testable on a host machine without needing an ESP32 or a real appliance.

=== Interface

.components/subzero_protocol/protocol.h (elided)
[source,cpp]
----
namespace esphome::subzero_protocol {
  // Zero-copy variants used in production: deserialize in ArduinoJson's
  // in-situ mode, consuming (mutating) the input buffer.
  FridgeState     parse_fridge_in_place(std::string &json, bool capture_keys = true);
  DishwasherState parse_dishwasher_in_place(std::string &json, bool capture_keys = true);
  RangeState      parse_range_in_place(std::string &json, bool capture_keys = true);

  // Copying overloads - same results, input preserved (used by tests).
  FridgeState     parse_fridge(const std::string &json, bool capture_keys = true);
  DishwasherState parse_dishwasher(const std::string &json, bool capture_keys = true);
  RangeState      parse_range(const std::string &json, bool capture_keys = true);
}
----

Each state struct has:

* `bool valid` - `false` on malformed JSON, unknown message format, or `status != 0`
* `bool is_poll` - `true` for `{"status":0,"resp":{...}}` poll responses, `false` for `{"seq":N,"props":{...}}` pushes. The hub uses this to set `poll_ok` only on real poll responses (see <<reconnect-logic>>).
* `std::optional<int> status` - the raw top-level `status` value, captured even when the parse is rejected. The hub uses it to detect `status: 302` (pairing revoked) without re-scanning the raw buffer.
* `bool lacking_properties` - `true` when the appliance rejected the poll verb with `{"status":1,"resp":{}}` ("lacking properties"); triggers the hub's `get_async` -> `get` fallback.
* `std::vector<std::string> data_keys` - top-level keys in the data object, in order. Only collected when `capture_keys` is true; the hub passes its debug-mode flag so the allocation is skipped on the normal poll path.
* `CommonFields common` - shared appliance metadata (model, uptime, diagnostic_status, version struct, etc.)
* Appliance-specific `std::optional<T>` fields for each sensor key

The per-appliance hubs check `s.valid`, `s.<field>.has_value()` and publish to entities - no `json::parse_json` calls, no `is<T>()` / `as<T>()` key fishing.

=== Running the Host Tests

A CMake + GoogleTest harness at `tests/cpp/` compiles the parser, message buffer, commands, and the full hub state machine on the host and runs 200+ tests (fixture-based parser tests plus targeted behavior tests for the buffer, dispatch, and reconnect logic). Fixtures live at `tests/fixtures/` - one `.json` + one `.expected.json` per captured BLE payload.

[source,bash]
----
cd tests/cpp
cmake -S . -B build
cmake --build build -j
ctest --test-dir build --output-on-failure
----

First run fetches ArduinoJson, GoogleTest, and nlohmann/json via CMake's `FetchContent` (~20 seconds). Subsequent runs compile in under a second and tests complete in milliseconds. The CI workflow runs the same commands on every push and PR (cached).

To add a fixture for a new appliance variant: turn on *Debug Mode*, concatenate the `Response[N/M]:` lines from the ESPHome logs (grep `Response\[` and stitch in order) into `tests/fixtures/<name>.json`, then record the expected parser output in `tests/fixtures/<name>.expected.json`. The test discovery picks up new pairs automatically.

== Debug Mode

Each appliance exposes a *Debug Mode* switch and a *Log Debug Info* button in Home Assistant (both under diagnostic entities).
When debug mode is enabled, the full response payload is logged at `INFO` level in chunked `Response[N/M]:` lines (chunked at 400 bytes to fit the per-line log budget) and all top-level response keys are logged under the `szg-debug` tag - making it easy to identify what data an untested appliance sends.
Grep for `Response\[` and concatenate the payloads in order to reassemble the full JSON.

IMPORTANT: Debug-mode payload logging is intentionally gated behind the switch. Normal operation never logs raw BLE bytes, because any non-UTF-8 byte in a log message crashes Home Assistant's protobuf decoder (`LogEntryResponse` requires valid UTF-8) and HA caches the bad event into a restart loop. When debug mode IS on, bytes outside the printable-ASCII range are replaced with `?` as a safety net.

The *Log Debug Info* button is a one-press shortcut: it turns on debug mode and lands a fresh poll. It always forces a BLE disconnect - `auto_connect: true` brings the connection back via `post_bond` + `subscribe`, which lands a fresh full-state get_async response on D6 in ~10-15 seconds. We use disconnect uniformly because fw 8.5 sessions cannot be refreshed in-place (the official Sub-Zero app does the same), and the extra ~10s wait on fw 2.27 keeps the code path simple.

=== Capturing a Full Poll

The procedure is the same for every appliance type:

. Press *Log Debug Info*.
. Wait ~10-15s for the reconnect cycle to complete and the next D6 poll to land.
. Grep `Response\[` in the ESPHome logs, concatenate the chunks in order to reassemble the full JSON.
. Optionally *interact with the appliance* (open/close doors, change set temp, toggle the light) to capture push notifications too. Each change triggers a small push on D6 with just the changed field - useful for confirming a sensor binding is correct.
. Turn *Debug Mode* off when done.
