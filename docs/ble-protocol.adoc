== GATT Service Structure

All Sub-Zero appliances expose a single custom GATT service:

[cols="1,2"]
|===
| Item | Value

| *Service UUID* | `E20A39F4-73F5-4BC4-A12F-17D1AD07A961`
| *D4 Characteristic* | `08590F7E-DB05-467E-8757-72F6FAEB13D4`
| *D5 Characteristic* | `08590F7E-DB05-467E-8757-72F6FAEB13D5`
| *D6 Characteristic* | `08590F7E-DB05-467E-8757-72F6FAEB13D6`
| *D7 Characteristic* | `08590F7E-DB05-467E-8757-72F6FAEB13D7`
| *D8 Characteristic* | `08590F7E-DB05-467E-8757-72F6FAEB13D8`
|===

The number of visible characteristics varies by appliance type:

* **Dishwashers & Ranges** expose *D4 and D5* only.
* **Refrigerators** expose *all five* (D4-D8).

=== Channel Roles

The characteristics serve different roles depending on the appliance type:

[cols="1,2,2"]
|===
| Channel | Dishwashers & Ranges | Refrigerators

| *D4*
| Open (unencrypted) diagnostic channel
| Open channel - responds to `get_async` with basic diagnostic data; also mirrors D5 push notifications with `msg_types: 8`

| *D5*
| Encrypted command/response channel - handles `unlock_channel`, `get_async`, `display_pin`, and delivers push notifications
| Encrypted channel - delivers push notifications (`msg_types: 2`) for state changes; also responds to `get_async` after a successful `unlock_channel`, but the unlock session expires after some time (possibly hours), after which command responses stop until a fresh unlock is performed

| *D6*
| _Not present_
| Encrypted channel - accepts writes, no observed responses

| *D7*
| _Not present_
| Open channel - accepts writes, no observed responses

| *D8*
| _Not present_
| Open channel - accepts writes, no observed responses
|===

NOTE: Firmware versions are model-dependent, not simply split by appliance category. The Sub-Zero 313 fridge and Wolf SO3050PESP wall oven both run API 5.4 / fw 8.5, while the Sub-Zero IW30R wine fridge, Cove dishwashers, and Wolf ranges run API 5.5 / fw 2.27.

== Connection & Authentication Flow

The full handshake follows these steps (shown for dishwashers/ranges where D5 is the command channel):

[source]
----
  ESP32                           Appliance
    |                                 |
    +---- BLE Connect --------------->|
    |                                 |
    |<--- Service Discovery ----------|  (only D4 visible initially)
    |                                 |
    +---- Request Encryption -------->|  esp_ble_set_encryption(MITM)
    |                                 |
    |<--- Passkey Request ------------|  (6-digit PIN shown on appliance display)
    +---- Passkey Reply ------------->|
    |                                 |
    |<--- Bond Complete (KEY_EVTs) ---|
    |                                 |
    +---- GATT Cache Refresh -------->|  esp_ble_gattc_cache_refresh()
    |                                 |
    |<--- Service Re-discovery -------|  (D5 now visible!)
    |                                 |
    +--- Subscribe D5 (indications) ->|  CCCD = 0x0200
    +--- Subscribe D4 (indications) ->|
    |                                 |
    +---- unlock_channel ------------>|  {"cmd":"unlock_channel","params":{"pin":"XXXXXX"}}\n
    |<--- PIN response ---------------|  {"status": 0, "resp": {"pin": "XXXXXX"}}
    |                                 |
    +---- get_async ----------------->|  {"cmd":"get_async"}\n
    |<--- Full JSON state ------------|  (fragmented across indications)
    |                                 |
----

=== Refrigerator Differences

Fridges follow the same connection and bonding flow, but command/response routing differs:

* *D4 polling:* `get_async` on *D4* works without an unlock and returns basic diagnostic data (model, door status, uptime) but not the full sensor state.
* *D5 polling:* `get_async` on *D5* returns the full sensor state (set temperatures, ice maker status, sabbath mode, wine zones, etc.) *after a successful `unlock_channel`*. However, the unlock session expires roughly 20-25 minutes after unlock. After expiry, writes to D5 are still accepted (GATT write status 0) but produce no indication response. Re-sending `unlock_channel` does *not* restore the session on some appliances (wall ovens, fridges with fw 8.5) - a full BLE reconnect is required. The integration detects this by tracking poll-response timeouts independently from push activity; see link:advanced.adoc#reconnect-logic[Reconnect Logic] for details.
* *Push notifications:* Full sensor data also arrives via *D5 push notifications* triggered by state changes on the appliance. Push notifications continue even after the unlock session expires (they are CCCD-based, not session-based).
* *Push mirroring:* Fridges send each push notification on *both* D4 (`msg_types: 8`) and D5 (`msg_types: 2`) simultaneously with identical `props` content.

=== Key Protocol Details

BLE Security:: The appliance requires *Legacy Pairing with MITM protection and bonding*.
The ESP32 must be configured with `IO_CAP_IN` (keyboard-only) so the user can enter the 6-digit passkey displayed on the appliance.
+
NOTE: Sub-Zero's BLE module seems to only support Legacy Pairing, *not* BLE Secure Connections (LESC). Do not set `ESP_LE_AUTH_REQ_SC_MITM_BOND` - it causes pairing to silently fall back to "Just Works" with no passkey exchange.
After the first successful bond, the keys are stored and reconnection is automatic.
If the bond becomes stale (e.g. appliance-side timeout overnight), the integration detects repeated auth failures (3 attempts) and automatically clears the stale bond, then re-pairs from scratch.

Command Framing:: Every command written to D5 *must* end with a newline character (`\n`).
The appliance's serial parser silently discards any command without it.

Response Fragmentation:: The appliance sends JSON responses as BLE indications, fragmented into chunks.
Fridges fragment at ~40 bytes per indication on both D4 and D5 (30-45 fragments for a full poll response), while dishwashers and ranges use larger ~244 byte fragments on D5 (7-8 fragments for a ~1600-1736 byte response).
The receiver must buffer and reassemble these fragments by tracking JSON brace depth (`{` / `}`) to detect message completion.

Push Notifications:: In addition to polled responses (`{"status":0,"resp":{...}}`), appliances send real-time push notifications for state changes (e.g. door open/close, temperature change, light toggle).
These use the format `{"seq":N,"props":{...},"msg_types":M}` where `msg_types` indicates the channel:
+
* `msg_types: 2` - D5 (encrypted channel)
* `msg_types: 8` - D4 (open channel, fridges only)
+
The parser handles both poll and push formats, extracting sensor data from `"resp"` (polls) or `"props"` (pushes).

D5 Discovery Timing:: On refrigerators, the encrypted D5 characteristic is *not visible* in the GATT database until after bonding completes.
The implementation must refresh the GATT cache (`esp_ble_gattc_cache_refresh`) and re-run service discovery after encryption is established.
The code polls up to three times with delays to handle timing variations across different appliance models.
+
NOTE: Cove dishwashers expose D5 in the GATT database *before* bonding, but still require BLE-level encryption for writes (GATT status 5 = insufficient authentication without it). The code always requests encryption regardless of when D5 is discovered.

MTU:: Negotiated to 517 bytes, though the appliance still fragments at ~40 bytes per indication (fridges) or ~244 bytes (dishwashers/ranges).

ACL Fragment Drops (ESP32-S3):: A bug in the ESP-IDF Bluedroid HCI layer causes ACL fragment reassembly failures during rapid bursts of indications.
This manifests as `BT_HCI: ACL packet too short` and `reassemble_and_dispatch found unfinished packet` errors.
When a fragment is dropped, the JSON reassembly buffer is left with incomplete data, and in some cases trailing non-UTF-8 bytes (stale memory) leak into the buffer.
This seems to only affect appliances using 40-byte indication fragments (API 5.4 / fw 8.5, e.g. SO3050PESP wall oven, 313 fridge, DEU2450R under-counter fridge, possibly others).
Appliances using 244-byte fragments (API 5.5 / fw 2.27, e.g. DF36450GSP range, DW2450 dishwasher, IW30R wine storage) are unaffected since their push notifications fit in a single indication.
+
The integration mitigates the corruption in three layers:
+
. **BLE notify handlers:** when a new chunk starts with `{` while the buffer has an unbalanced previous message (brace depth != 0), the previous message was truncated - discard the buffer before appending the new data. Logs `ACL corruption: discarding N bytes of partial data`.
. **parse_json scripts:** trim the buffer to exact JSON extent (first `{` through matching `}`), then *parse first, publish second*. If the parser rejects the message (malformed, `status!=0`, or a nested object like `{"notif_seq":N,...}` left over from a truncated response), it is NOT published to the `Raw JSON` text sensor. ESPHome text sensors serialize as protobuf `string`, which must be valid UTF-8 - raw binary crashes Home Assistant's protobuf decoder, and because HA caches the bad state, causes a restart loop.
. **Session-alive signal:** when layer 1 fires, `poll_in_flight` and `poll_no_response` are cleared. The appliance DID respond (data was flowing, we just couldn't reassemble it), so it should not count as an unanswered poll.
. **Sticky `acl_corruption_seen` flag:** layer 3 alone is not enough. On fw 8.5 appliances the *next* poll after ACL corruption silently fails - the session is effectively wedged by the corrupted exchange. That silent poll would re-trigger unlock-expiry detection, reconnect, and the ACL bug hits the next fresh session the same way - a reconnect loop forever. The sticky flag is set on any ACL corruption and permanently disables the unlock-expiry reconnect logic for that device. Zombie detection (15 min of total silence) still fires if the connection truly dies. Cleared only on ESP reboot. The side effect is the appliance stops getting poll-only sensors refreshed (uptime, model, version tree, sabbath_on) - but that was already unreliable since those fields only arrive in the full response that keeps getting corrupted.
+
Even with all three layers, the *first push notification in a burst* is typically lost at the appliance side - for example, `cav_gourmet_mode_on` during gourmet mode activation. That fragment truly never arrives; no amount of buffer handling can recover it. Subsequent state changes in the same session are received correctly. And on appliances whose full-state response reliably hits the bug (wall ovens, 313 fridge), the "poll-only" sensors (model, uptime, serial, sabbath_on, version tree) may never populate - they only arrive in the full response that keeps getting corrupted.

== Command Reference

Commands are JSON objects terminated with `\n`. The target characteristic depends on the appliance type and command.

=== Dishwashers & Ranges

All commands are written to *D5*. Responses arrive as D5 indications.

[cols="1,2,2"]
|===
| Command | Payload | Response

| `display_pin` | `{"cmd":"display_pin","params":{"duration": 30}}\n` | Displays PIN on appliance for 30 seconds
| `unlock_channel` | `{"cmd":"unlock_channel","params":{"pin":"XXXXXX"}}\n` | `{"status": 0, "resp": {"pin": "XXXXXX"}}`
| `get_async` | `{"cmd":"get_async"}\n` | Full appliance state as JSON (see <<Example Responses>>)
|===

=== Refrigerators

Fridges use different channels for different operations:

[cols="1,1,2,2"]
|===
| Command | Channel | Payload | Response

| `get_async` | *D4* | `{"cmd":"get_async"}\n` | Basic diagnostic data, no unlock needed (see <<D4 Response>>)
| `unlock_channel` | *D5* | `{"cmd":"unlock_channel","params":{"pin":"XXXXXX"}}\n` | `{"status": 0, "resp": {"pin": "XXXXXX"}}` (after successful unlock)
| `get_async` | *D5* | `{"cmd":"get_async"}\n` | Full appliance state (requires prior `unlock_channel`; session expires after some time)
| `display_pin` | D5 | `{"cmd":"display_pin","params":{"duration": 30}}\n` | Displays PIN on appliance display
|===

NOTE: D5 command responses require an active unlock session. After the session expires (~20-25 minutes), `get_async` writes to D5 are accepted but produce no response. On fridges/wall ovens with fw 8.5, re-sending `unlock_channel` does not restore the session - a full BLE reconnect is required. Push notifications on D5 continue regardless of session state, which is why expiry detection has to track poll responses specifically rather than relying on "any D5 activity". See link:advanced.adoc#reconnect-logic[Reconnect Logic].

=== D4 `get_async` Response (Refrigerators)

Fridges respond to `get_async` on D4 with basic diagnostic data. This is a smaller response than what dishwashers/ranges return on D5.

.French-door fridge/freezer (313) - ~403 bytes
[%collapsible]
====
[source,json]
----
{
  "status": 0,
  "resp": {
    "diagnostic_status": "0x00000003111",
    "service": {},
    "version": {
      "api": "5.4",
      "fw": "8.5",
      "appliance": "main: 3.12.0; conf: 99.3.10; uim: N/A"
    },
    "appliance_name": "SubZeroCAT",
    "appliance_type": "1.1.1.12",
    "appliance_model": "313",
    "ref_door_ajar": false,
    "frz_door_ajar": false,
    "uptime": "12:39:03",
    "device_wlan_id": "000680229d63"
  }
}
----
====

NOTE: Keys like `ref_set_temp`, `frz_set_temp`, `ice_maker_on`, and `sabbath_on` are *not* included in D4 responses. These are available via D5 `get_async` (with an active unlock session) or D5 push notifications when the appliance state changes.

=== D5 Push Notification (Refrigerators)

Sensor data from fridges arrives as push notifications on D5. Each push contains only the fields that changed.

.Door opened
[source,json]
----
{"msg_types": 2, "seq": 75, "timestamp": "2025-01-23T09:43:54.901", "props": {"ref_door_ajar": true}}
----

.Temperature change
[source,json]
----
{"msg_types": 2, "seq": 80, "props": {"ref_set_temp": 38}}
----

The combined set of keys observed across D4 polls and D5 push notifications for fridge/freezer units includes: `ref_door_ajar`, `frz_door_ajar`, `ref_set_temp`, `frz_set_temp`, `ice_maker_on`, `max_ice_on`, `night_ice_on`, `sabbath_on`, `service_required`, `water_filter_gal_remaining`, `water_filter_pct_remaining`, `air_filter_pct_remaining`, `air_filter_on`, `appliance_model`, `uptime`, and others. Refrigerator-only units (no freezer) will not have `frz_*` or `ice_maker_*` keys. Wine storage units include `wine_door_ajar`, `wine_set_temp`, `wine2_set_temp`, and `wine_temp_alert_on`.

IMPORTANT: Fridge firmware exposes only setpoints (`*_set_temp`) - no actual/measured compartment temperatures are published over BLE. This was confirmed empirically against an IW30R (fw 2.27) by capturing full `get_async` responses across a setpoint change that would drive the compressor; no `*_cur_temp`, `*_actual_temp`, or plain `*_temp` keys appeared in any response. The derived boolean `wine_temp_alert_on` shows the firmware tracks the real value internally, but it is not published. (Wolf ranges, by contrast, do expose `cav_temp` and `cav_probe_temp` as actual measurements distinct from their `*_set_temp` counterparts.)

=== GATT Handle Map (Refrigerator)

For reference, the GATT handle layout observed on a Sub-Zero 313 fridge/freezer:

[cols="1,1,1"]
|===
| Characteristic | Handle | Notes

| D5 | 3 | Push notifications, encrypted
| D6 | 7 | Encrypted, no observed responses
| D4 | 11 | Open channel, responds to `get_async`
| D7 | 15 | Open, no observed responses
| D8 | 19 | Open, no observed responses
|===

NOTE: Handle numbers are assigned by the appliance firmware and may vary across models. The implementation discovers handles dynamically by UUID suffix, not by hardcoded values.
